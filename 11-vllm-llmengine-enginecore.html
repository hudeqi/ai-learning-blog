1. 为什么从这里读？

前几篇已经把 vLLM 的三个核心组件读完了：

  • BlockPool（Ep.7）：物理 KV Cache 块的分配与回收
  • KVCacheManager（Ep.8/9）：逻辑→物理的映射层
  • Scheduler（Ep.10）：决定每一步推理谁来跑

但有一个问题一直没有回答：这些组件是被谁调起来的？ 用户调用 llm.generate() 之后，请求是怎么流进 Scheduler 的？GPU 跑完之后，token 又是怎么流回给调用方的？

答案就在 LLMEngine 和 EngineCore 这两个文件里。

一句话定位：LLMEngine 是对外暴露的接口层（门面），EngineCore 是实际驱动调度和执行的内核（心脏）。两者合在一起，才是 vLLM 推理引擎的"主控程序"。

2. 整体架构：两层设计

用户代码 │ │ llm.generate("prompt") ▼ ┌─────────────────────────────────────────────────┐ │ LLMEngine │ │ ┌──────────────┐ ┌──────────────────────────┐ │ │ │InputProcessor│ │ OutputProcessor │ │ │ │ (文本→token) │ │ (token→文本, detokenize) │ │ │ └──────────────┘ └──────────────────────────┘ │ │ │ │ engine_core = EngineCoreClient ──────────────┐ │ └───────────────────────────────────────────────┼──┘ │ ┌───────────────────────▼──┐ │ EngineCore │ │ ┌──────────────────────┐ │ │ │ Scheduler │ │ │ │ (你在 Ep.10 读过) │ │ │ └──────────────────────┘ │ │ ┌──────────────────────┐ │ │ │ ModelExecutor │ │ │ │ (GPU forward pass) │ │ │ └──────────────────────┘ │ └───────────────────────────┘

关键分工：

层类职责文件
门面层LLMEngine接收用户请求、预处理、后处理、对外暴露 add_request/stepv1/engine/llm_engine.py
内核层EngineCore驱动 Scheduler + ModelExecutor，是实际的调度执行循环v1/engine/core.py
通信层EngineCoreClient单进程时直接调用，多进程时通过 ZMQ 跨进程通信v1/engine/core_client.py
V1 的重要变化：vLLM V1 把 LLMEngine 标注为 "Legacy LLMEngine for backwards compatibility"。真正的新架构入口是 AsyncLLM（对应 API Server 的异步场景），LLMEngine 是为了兼容 V0 的离线推理接口保留的。但它的内部已经全面切换到 V1 的 EngineCore，所以读 LLMEngine 依然是理解整个链路的最佳切入点。

3. LLMEngine：门面层

3.1 初始化：组装三大模块

# llm_engine.py __init__ 核心片段（简化）
self.input_processor  = InputProcessor(vllm_config, renderer)
self.output_processor = OutputProcessor(renderer.tokenizer, ...)
self.engine_core      = EngineCoreClient.make_client(
    multiprocess_mode=multiprocess_mode, ...)

三个模块，职责清晰：

  • InputProcessor：把用户传进来的文字/token_ids 转成 EngineCoreRequest（内部请求格式），做长度校验、参数校验、多模态处理
  • OutputProcessor：把 EngineCoreOutputs（token ids）逆向转成文字（detokenize），并组装成 RequestOutput 返回给用户
  • EngineCoreClient：EngineCore 的代理，单进程直接调用，多进程通过 ZMQ 通信

3.2 add_request：请求入队

def add_request(self, request_id, prompt, params, ...):
    # 1. 文字 → EngineCoreRequest（tokenize + 校验）
    request = self.input_processor.process_inputs(...)
    
    # 2. 给 request_id 加随机后缀（防重复）
    self.input_processor.assign_request_id(request)
    
    n = params.n  # 用户要求生成几个结果
    if n == 1:
        self.output_processor.add_request(request, ...)
        self.engine_core.add_request(request)
    else:
        # n > 1：fan out，拆成 n 个 child request
        for idx in range(n):
            child_request = copy(request)
            self.engine_core.add_request(child_request)
n > 1 的 fan-out：当用户要求生成 n 个候选结果时（SamplingParams(n=3)），LLMEngine 会把原始请求 clone 成 n 个 child request，每个独立推理，最后通过 ParentRequest 合并输出。这也是为什么你在 Scheduler 里看到的请求都是单个的——fan-out 在更上层就做掉了。

3.3 step：一次推理心跳

def step(self) -> list[RequestOutput]:
    # 1. 从 EngineCore 取本轮输出
    outputs = self.engine_core.get_output()
    
    # 2. Detokenize：token_ids → 文字，组装 RequestOutput
    processed = self.output_processor.process_outputs(outputs.outputs, ...)
    
    # 3. 把因 stop strings 结束的请求通知 EngineCore 中止
    self.engine_core.abort_requests(processed.reqs_to_abort)
    
    # 4. 记录统计
    if self.logger_manager:
        self.logger_manager.record(...)
    
    return processed.request_outputs

step() 本身很薄：它不做调度，不做 GPU 执行，只做"拿结果、转文字、返回给用户"。真正的调度和执行都在 EngineCore 里。

4. EngineCore：心脏层

4.1 初始化：启动顺序非常关键

# core.py EngineCore.__init__ 核心启动顺序
1. model_executor = Executor(vllm_config)   # 加载模型权重到 GPU
2. kv_cache_config = _initialize_kv_caches() # GPU 显存 profiling → 分配 KV Cache
3. scheduler = Scheduler(vllm_config, kv_cache_config, ...)  # 创建调度器

为什么顺序重要？

  • 必须先加载模型权重，才能知道模型实际占用多少 GPU 显存
  • 用剩余显存才能计算最多能放多少 KV Cache blocks（determine_available_memory()）
  • 有了 kv_cache_config，Scheduler 才知道自己能管理多少 blocks
这里解答了一个常见问题：vLLM 启动时为什么那么慢？就是因为这个初始化流程——加载模型 + GPU profiling + 编译（如果开了 torch.compile）+ warmup，缺一不可。

4.2 step：EngineCore 真正的心跳

def step(self):
    if not self.scheduler.has_requests():
        return {}, False
    
    # 1. Scheduler 决定本轮跑谁、每个 request 分配多少 token slot
    scheduler_output = self.scheduler.schedule()
    
    # 2. 把 scheduler_output 扔给 GPU 异步执行
    future = self.model_executor.execute_model(scheduler_output, non_block=True)
    
    # 3. 获取 grammar bitmask（结构化输出约束，普通推理可忽略）
    grammar_output = self.scheduler.get_grammar_bitmask(scheduler_output)
    
    # 4. 等待 GPU 执行完，拿到 logits 之后采样
    model_output = future.result()
    if model_output is None:
        model_output = self.model_executor.sample_tokens(grammar_output)
    
    # 5. 处理推理过程中产生的 abort 请求
    self._process_aborts_queue()
    
    # 6. 用本轮输出更新 Scheduler 状态
    #    （KV Cache 释放/分配、请求状态迁移——你在 Ep.10 详细读过这里）
    engine_core_outputs = self.scheduler.update_from_output(
        scheduler_output, model_output)
    
    return engine_core_outputs, scheduler_output.total_num_scheduled_tokens > 0

把 Ep.10 里学的 Scheduler 放回这里，整个流程就通了：

EngineCore.step() │ ├─ scheduler.schedule() │ └─ 分配 KV Cache blocks，决定 prefill/decode 请求 │ ├─ model_executor.execute_model(scheduler_output) │ └─ GPU 跑 transformer forward pass（异步，non_block=True） │ ├─ future.result() ← 阻塞，等 GPU 完成 │ ├─ sample_tokens() ← 从 logits 采样，得到下一个 token │ └─ scheduler.update_from_output() └─ 更新 KV Cache 状态，标记 finished 请求

4.3 step_with_batch_queue：流水线并行优化

当 max_concurrent_batches > 1 时，EngineCore 会启用 batch queue 模式，实现调度和 GPU 执行的流水线并行：

普通模式（step）： [调度] → [GPU执行] → [采样] → [调度] → [GPU执行] → [采样] GPU idle 在调度期间 batch queue 模式（step_with_batch_queue）： [调度A] → [GPU执行A] [调度B] → [GPU执行B] [调度C] → [GPU执行C] ↑ 调度和 GPU 执行重叠，消除 pipeline bubble
工程亮点：这是 vLLM 针对 pipeline parallelism 场景专门做的优化。多 GPU 流水线并行时，GPU 之间需要传递中间激活值，batch queue 让下一个 batch 的调度在 GPU 传输期间就完成，避免空泡。

5. 一次推理的完整生命周期

把前几篇的内容全部串起来，看一次 llm.generate("hello") 是怎么走完整个系统的：

用户: llm.generate("hello") │ ▼ LLMEngine.add_request() │ InputProcessor.process_inputs() │ → tokenize: "hello" → [9891] │ → 校验长度、params │ → 打包成 EngineCoreRequest │ ▼ EngineCoreClient.add_request() │ ▼ EngineCore.scheduler.add_request() │ → 请求进入 WAITING 队列 │ ▼ ┌─────── 推理循环（run_busy_loop） ────────────────┐ │ │ │ EngineCore.step() │ │ │ │ │ ├─ scheduler.schedule() │ │ │ → WAITING → RUNNING │ │ │ → 分配 KV Cache blocks（BlockPool） │ │ │ → 构建 SchedulerOutput │ │ │ │ │ ├─ model_executor.execute_model() │ │ │ → 把 token_ids + KV Cache 地址传给 GPU │ │ │ → GPU 跑 attention + FFN forward │ │ │ → 返回 logits │ │ │ │ │ ├─ sample_tokens() │ │ │ → top-p/top-k/temperature 采样 │ │ │ → 得到下一个 token id（如 5911 = "world") │ │ │ │ │ └─ scheduler.update_from_output() │ │ → 如果遇到 EOS，标记 FINISHED │ │ → 释放 KV Cache blocks │ │ │ └──────────────────────────────────────────────────┘ │ ▼ LLMEngine.step() │ engine_core.get_output() │ output_processor.process_outputs() │ → detokenize: [5911] → "world" │ → 拼接成完整 text: "hello world" │ ▼ 用户拿到: RequestOutput(text="hello world")

6. 多进程模式：EngineCoreProc

当 VLLM_ENABLE_V1_MULTIPROCESSING=1 时，EngineCore 会运行在独立进程里（EngineCoreProc），通过 ZMQ socket 与 LLMEngine 通信。

6.1 为什么要多进程？

Python GIL 问题：LLMEngine 做 tokenize/detokenize 是 CPU 密集型操作，如果和 GPU 执行在同一个进程里，会争 GIL，导致 GPU 空等 CPU。

多进程方案：

进程 A（LLMEngine 进程） 进程 B（EngineCore 进程） │ │ │ InputProcessor（tokenize） │ Scheduler │ OutputProcessor（detokenize） │ ModelExecutor（GPU） │ │ │ ←─── ZMQ PUSH/PULL socket ──────────────→│ │ (msgpack 序列化) │ │ │ │ input_thread（接收 GPU 结果） │ input_thread（接收用户请求） │ output_thread（发送用户请求） │ output_thread（发送 GPU 结果）

6.2 run_busy_loop：EngineCore 的主循环

def run_busy_loop(self):
    while self._handle_shutdown():
        # 1. 从 input_queue 消费：ADD / ABORT / UTILITY 请求
        self._process_input_queue()
        # 2. 执行一步推理，把结果放进 output_queue
        self._process_engine_step()

input_thread 和 output_thread 各自用独立线程跑 ZMQ socket IO，与 GIL 无关（ZMQ 操作释放 GIL），CPU 处理和 GPU 执行真正并行。

6.3 启动握手（Handshake）

多进程模式下，EngineCore 进程启动时需要与 LLMEngine 进程完成握手，交换 ZMQ 地址信息：

# EngineCore 进程启动后：
1. 发送 HELLO 消息（带 identity）到 handshake_address
2. 等待 LLMEngine 返回 EngineHandshakeMetadata（包含 input/output socket 地址）
3. 连接 input/output socket
4. 发送 READY 消息，通知 LLMEngine 可以开始发送请求
这是经典的 ZMQ DEALER-ROUTER 握手模式。LLMEngine 侧是 ROUTER（可以同时和多个 EngineCore 通信），EngineCore 侧是 DEALER（有固定的 identity）。

8. 面试高频 Q&A

Q1：vLLM 的 LLMEngine 和 EngineCore 有什么区别？

A：LLMEngine 是对外暴露的门面，负责 tokenize/detokenize、参数校验、n>1 的 fan-out；EngineCore 是实际驱动推理的内核，持有 Scheduler 和 ModelExecutor，执行调度+GPU forward+采样的完整循环。两者通过 EngineCoreClient 解耦，可以跑在同一个进程里，也可以通过 ZMQ 跑在不同进程里（多进程模式下 GPU 利用率更高）。

Q2：vLLM 为什么要多进程？GIL 和 GPU 利用率有什么关系？

A：Python GIL 保证同一时刻只有一个线程执行 Python bytecode。tokenize/detokenize 是 CPU 密集型 Python 操作，如果和 GPU 推理在同一进程里，CPU 操作持有 GIL 时 GPU 就在空转。多进程模式下，LLMEngine 进程做 CPU 处理，EngineCore 进程专门跑 GPU，两者并行，GPU 利用率显著提升。

Q3：一个请求从 add_request 到最终拿到结果，经历了哪些关键步骤？

A：① InputProcessor 把文字 tokenize 并打包成 EngineCoreRequest → ② EngineCore.scheduler.add_request 把请求加入 WAITING 队列 → ③ EngineCore.step 里 Scheduler 调度到 RUNNING → ④ ModelExecutor 执行 GPU forward → ⑤ sample_tokens 采样出下一个 token → ⑥ update_from_output 更新状态，如果生成结束则 FINISHED → ⑦ OutputProcessor detokenize token_ids，组装 RequestOutput 返回给用户。

Q4：batch queue 模式是什么，解决了什么问题？

A：batch queue 是 step_with_batch_queue 里实现的流水线并行优化。普通模式下，调度（CPU）和 GPU 执行是串行的，调度期间 GPU 空转。batch queue 让 EngineCore 提前调度下一个 batch，使 CPU 调度和 GPU 执行重叠，消除 pipeline bubble。主要针对 pipeline parallelism（多 GPU 流水线）场景。

Q5：EngineCore 初始化为什么要按"加载权重 → profile 显存 → 分配 KV Cache"的顺序？

A：必须先把模型权重加载到 GPU，才能知道权重占用了多少显存（通过 determine_available_memory() profile）；用剩余显存才能计算最多能放多少 KV Cache blocks；有了 block 数量，Scheduler 才能初始化 BlockPool 和 KVCacheManager。顺序颠倒任何一步都会导致显存分配错误或 OOM。

9. 总结

本篇读完之后，vLLM 推理引擎的全貌已经呈现：

  • LLMEngine = 门面。三个成员：InputProcessor + OutputProcessor + EngineCoreClient。核心方法：add_request()（请求入队）+ step()（拿结果返回用户）。
  • EngineCore = 心脏。三件事：初始化（权重→profiling→KVCache→Scheduler）、step()（调度→GPU执行→采样→更新状态）、多进程通信（ZMQ + 独立 IO 线程）。
  • step() 的两种形态：单 batch 的 step() 和流水线并行的 step_with_batch_queue()，后者通过 batch queue 消除 pipeline bubble。
  • 多进程架构：EngineCoreProc 用 ZMQ 把 CPU 处理和 GPU 执行解耦到不同进程，彻底解决 GIL 瓶颈。
